### Enabling AOT Mode

When you set `aot: true` in the router options, the router operates in strict AOT mode:

- **No run-types loading**: The heavy `@mionkit/run-types` package is NOT loaded at all
- **Fail-fast validation**: If any route/hook is missing from the AOT cache, an `AOTCacheError` is thrown immediately
- **Smaller bundles**: With proper tree-shaking, run-types is excluded from production bundles

```typescript
// Enable strict AOT mode
await initMionRouter(routes, {
  aot: true,  // Strict AOT mode - run-types will NOT be loaded
});
```

::warning
When `aot: true` is set, ALL routes and hooks MUST be present in the AOT cache. If any are missing, the router will throw an `AOTCacheError` instead of falling back to JIT compilation.
::

## Client Usage

The mion client automatically loads AOT caches from `@mionkit/aot-caches` on initialization. For custom AOT caches, you can load them manually.

### Using Custom AOT Caches

::note
**AOT caches are less important on the client than on the server.** Browsers don't have the same restrictions as secure server environments (Cloudflare Workers, etc.), and the mion client can request any missing JIT function from the server on demand.

For larger APIs, it's often **better to NOT load AOT caches** in the client initially. This reduces bundle size and lets the client fetch only the JIT functions it actually needs.
::

If you still want to preload custom AOT caches (for offline support or reduced latency), load them before initializing the client:

## Secure Environments (Cloudflare Workers, etc.)

AOT compilation is essential for running mion in secure environments that restrict dynamic code execution.

### Why AOT is Required

Environments like Cloudflare Workers, Deno Deploy, and some enterprise environments prohibit:
- `eval()`
- `new Function()`
- Other forms of dynamic code generation

mion's JIT compilation normally uses `new Function()` to generate validation and serialization functions. AOT compilation pre-generates these functions as static code, eliminating the need for runtime code generation.

## Cache Exports Reference

Each AOT package exports the following:

| Export | Type | Description |
|--------|------|-------------|
| `routerCache` | `MethodsCache` | Pre-compiled router method metadata |
| `jitFnsCache` | `PersistedJitFunctionsCache` | Pre-compiled JIT functions |
| `pureFnsCache` | `PersistedPureFunctionsCache` | Pre-compiled pure helper functions |

### Missing Functions in Cache

If some functions are missing from the AOT cache, ensure your build script includes all routes:

```bash
# Use the exact same entry point as production
npx mion-build-aot \
  --dir ./packages/my-api-aot \
  --start-server-script ./dist/cjs/server.js
```

### AOTCacheError

When running in AOT mode (`aot: true`), you may encounter `AOTCacheError` if routes or hooks are missing from the cache:

```
AOTCacheError: Route/hook "users/getUser" not found in AOT cache.
Regenerate AOT caches using 'mion-build-aot' command.
```

**Solutions:**

1. **Regenerate AOT caches**: Run `npx mion-build-aot` to rebuild the caches with all current routes
2. **Check route registration**: Ensure all routes are registered before the AOT build runs
3. **Verify cache loading order**: Make sure `addAOTCaches()` is called before `initMionRouter()`

::note
If you don't need strict AOT mode, you can omit the `aot: true` option. The router will then fall back to JIT compilation for any missing routes (requires `@mionkit/run-types` to be available).
::
