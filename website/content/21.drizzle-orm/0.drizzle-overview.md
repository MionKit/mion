The package `@mionjs/drizzle` auto generates Drizzle ORM table schemas directly from TypeScript types.
Unlike `drizzle-zod` or other drizzle plugins that generates Zod schemas FROM drizzle tables, this package works in the **opposite direction**: it auto-generates drizzle table configurations FROM TypeScript types while allowing optional overrides.

::note
Column types are derived from Typescript types and [`@mionjs/type-formats`](../run-types/type-formats) - when you use format types like `FormatUUIDv7`, `FormatEmail`, `FormatInteger`, etc., the package automatically selects the most appropriate database column type for each database.
::

::tip
**Quick Tip:** 
This package is designed to work with [flat types that reflect the columnar structure of a database table]{.text-highlighted}, not deeply nested JavaScript objects with relationships. 
Use foreign key IDs instead of embedded entity references. Nested objects are stored as JSON columns.
::

## How To Use

#### Primary and Foreign Keys

Primary keys and foreign keys should always be defined in the tableConfig. This gives you full control over key constraints and relationships:

::tip
**Quick Tip:** 
Types should only contain information related to type and format, this will be used for validation and serialization.
Any columns asn SQL logic like primary, foreign keys, indexes, etc. should be defined in the tableConfig override.
::

#### Length Buffer

The `lengthBuffer` option multiplies the `maxLength` defined in custom string formats (`FormatString`) to provide extra capacity for the database column. This is useful when you want to allow for some flexibility in the data length prevent resizing in the future, but still want to validate the correct length at runtime.

::note
The `lengthBuffer` only applies to custom `FormatString` types with a `maxLength` constraint. Well-known formats like `FormatEmail`, `StrURL`, and `FormatDomain` use their standard maximum lengths without any buffer.
::

```ts
// Custom string format with maxLength
type Username = FormatString<{maxLength: 50}>;

interface User {
    id: FormatUUIDv7;
    username: Username;
}

// Default lengthBuffer is 1.5
const users = toDrizzlePGTable<User>('users');
// username with maxLength 50 → varchar(75)

// Custom lengthBuffer of 2.0
const users = toDrizzlePGTable<User>('users', undefined, {lengthBuffer: 2.0});
// username with maxLength 50 → varchar(100)
```

#### Table Config Validation

The function validates that your tableConfig matches the TypeScript type

```ts
interface User {
    id: string;
    name: string;
}

// ❌ Error: Column "email" exists in tableConfig but not in type "User"
const users = toDrizzlePGTable<User>('users', {
    id: uuid('id').primaryKey(),
    email: text('email'), // This property doesn't exist in User!
});
```

## Important Notes

#### Import Format Types Correctly

::info
When using format types from `@mionjs/type-formats`, you must use **regular imports** (not `import type`) for the runtime type metadata to be preserved.
::

```ts
// ✅ Correct - regular import preserves metadata
import {FormatUUIDv7, FormatEmail} from '@mionjs/type-formats/StringFormats';

// ❌ Wrong - type import strips metadata
import type {FormatUUIDv7, FormatEmail} from '@mionjs/type-formats';
```

#### Nested Objects vs Foreign Keys

Nested objects are stored as JSON. Use foreign key IDs for entity references:

```ts
// ✅ Good: Profile is a value object - stored as JSON
interface User {
    id: string;
    profile: {bio: string; avatar: string};
}

// ❌ Bad: Don't embed entire entities
interface Book {
    id: string;
    owner: User; // This stores the entire User as JSON!
}

// ✅ Good: Use foreign key ID instead
interface Book {
    id: string;
    ownerId: string; // Reference by ID
}
```
