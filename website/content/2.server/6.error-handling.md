mion uses `RpcError` to handle all errors in routes and hooks. The key principle is: **return errors instead of throwing them** to maintain full type safety on the client.

## Return Errors (Recommended)

**Always return errors instead of throwing them.** Returned errors are part of the function signature, giving the client full type information about possible error types and their data:

```ts
import {RpcError} from '@mionkit/core';
import {Routes, route} from '@mionkit/router';

type User = {id: string; name: string};

const routes = {
    // ✅ RECOMMENDED: Return error - client gets full type information
    getUser: route((ctx, userId: string): User | RpcError<'user-not-found', {userId: string}> => {
        const user = findUser(userId);
        if (!user) {
            return new RpcError({
                type: 'user-not-found',
                publicMessage: 'User not found',
                errorData: {userId}, // Typed error data
            });
        }
        return user;
    }),
} satisfies Routes;
```

The client can now handle errors with full type safety:

```ts
const {data, error} = await routes.getUser('123').call();

if (error) {
    // TypeScript knows error.type is 'user-not-found'
    // and error.errorData is {userId: string}
    console.log('User not found:', error.errorData?.userId);
    return;
}

// TypeScript knows data is User here
console.log('Found user:', data.name);
```

## Use `error.type` to Identify Errors

::warning
**HTTP status codes are irrelevant in mion's RPC architecture.** Always use `error.type` to identify and handle errors.
::

In RPC-style APIs, the HTTP status code is just a transport detail. The `error.type` property is the discriminator that identifies the error:

```ts
// ✅ Correct: Use error.type
if (error.type === 'user-not-found') {
    showUserNotFoundMessage();
} else if (error.type === 'not-authorized') {
    redirectToLogin();
}

// ❌ Wrong: Don't rely on HTTP status codes
if (response.status === 404) { ... } // Don't do this!
```

## Throwing Errors (Discouraged)

::warning
**Thrown errors lose type information.** Only throw errors for truly unexpected situations (bugs, infrastructure failures).
::

Thrown errors are treated as unexpected errors and only expose `publicMessage` to the client. The error type is not part of the function signature:

```ts
const routes = {
    // ❌ DISCOURAGED: Thrown error - client loses type information
    getUser: route((ctx, userId: string): User => {
        const user = findUser(userId);
        if (!user) {
            // Client cannot know about this error type at compile time
            throw new RpcError({
                type: 'user-not-found',
                publicMessage: 'User not found',
            });
        }
        return user;
    }),
} satisfies Routes;
```

## Error Response Format

Returned and thrown errors are serialized differently in the response body:

- **Returned errors** (strongly typed): Appear under the route id, preserving type information
- **Thrown errors** (untyped): Collected in the `@thrownErrors` field

```json
{
    "getUser": {
        "type": "user-not-found",
        "publicMessage": "User not found",
        "errorData": {"userId": "123"}
    },
    "@thrownErrors": {
        "someHook": {
            "type": "unknown-error",
            "publicMessage": "Unexpected error"
        }
    }
}
```

::note
The mion client abstracts this away — you simply access `error` or `errors` from the result object. The internal `message` property is never sent to the client.
::

## Multiple Error Types

Routes and hooks can return multiple error types. The client receives a discriminated union of all possible errors:

```ts
import {RpcError} from '@mionkit/core';
import {Routes, route} from '@mionkit/router';

type User = {id: string; name: string};

const routes = {
    updateUser: route((
        ctx,
        userId: string,
        data: {name: string}
    ): User | RpcError<'user-not-found'> | RpcError<'invalid-name', {reason: string}> => {
        const user = findUser(userId);
        if (!user) {
            return new RpcError({type: 'user-not-found', publicMessage: 'User not found'});
        }
        if (!isValidName(data.name)) {
            return new RpcError({
                type: 'invalid-name',
                publicMessage: 'Invalid name',
                errorData: {reason: 'Name must be at least 2 characters'},
            });
        }
        return updateUser(userId, data);
    }),
} satisfies Routes;
```

## Run On Error Hooks

Hooks with `runOnError: true` execute even when previous hooks or routes return/throw errors. Useful for logging or cleanup:

```ts
import {Routes, hook} from '@mionkit/router';

const routes = {
    logger: hook(
        (ctx): void => {
            const errors = ctx.request.internalErrors;
            if (errors.length > 0) {
                console.error('Request errors:', errors);
            }
        },
        {runOnError: true}
    ),
    // ... other routes
} satisfies Routes;
```

## Type Reference

### RpcError

```ts
class RpcError<ErrType extends string, ErrData = any> extends Error {
    /** Error type identifier - use this to identify errors */
    readonly type: ErrType;
    /** Public message sent to client */
    readonly publicMessage: string;
    /** Typed error data accessible on client */
    readonly errorData?: ErrData;
    /** Optional HTTP status code (transport detail, not for error identification) */
    statusCode?: number;
    /** Unique error ID (auto-generated if RouterOptions.autoGenerateErrorId is true) */
    readonly id?: string | number;
}
```
