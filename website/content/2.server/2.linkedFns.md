LinkedFns or `(Middleware functions)` are executed before or after a route gets executed. We call them linkedFns because they are functions that are chained to create the full [Execution Chain](/server/execution-chain) of a route.

LinkedFns are useful when a route might require some extra data like authorization, filters or some extra processing like body parsing, logging, etc...

The first parameter is always the [Call Context](/server/call-context), the rest of parameters are remote parameters that get deserialized and validated before the linkedFn gets executed.

## Raw LinkedFns

In case we need to access the raw or native underlying request and response, we must use a [`rawLinkedFn`](#rawlinkedFndef).

These are linkedFns that receive the `CallContext`, `RawRequest`, `RawResponse` and `RouterOptions`, but can't receive any remote parameters or return any data. `Raw LinkedFns` can only modify the CallContext and return or throw errors.

Raw LinkedFns are useful to extend the router's core functionality, i.e: The router internally uses a `Raw LinkedFn` for body serialization.

## Force Run On Errors

When there is an error in a route or linkedFn the rest of linkedFns are not executed unless `runOnError` is set to `true`.
This is useful for some linkedFns like a logger that needs to be executed after any other linkedFn and log all the errors or request data.
