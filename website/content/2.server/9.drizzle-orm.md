mion provides `@mionkit/drizzle` to auto-generate Drizzle ORM table schemas from TypeScript types and **mion Type Formats**.
Unlike `drizzle-zod` which generates Zod schemas FROM drizzle tables, this package works in the **opposite direction**: it auto-generates drizzle table configurations FROM TypeScript types while allowing optional overrides.

::note
The key feature is that column types are derived from [`@mionkit/type-formats`](../type-formats/overview) - when you use format types like `StrUUIDv7`, `StrEmail`, `NumInteger`, etc., the package automatically selects the most appropriate database column type for each database.
::

::tip
**Quick Tip:** 
This package is designed to work with [flat types that reflect the columnar structure of a database table]{.text-highlighted}, not deeply nested JavaScript objects with relationships. 
Use foreign key IDs instead of embedded entity references. Nested objects are stored as JSON columns.
::

## Type Mappings

#### Primitive Types

| TypeScript Type | PostgreSQL | MySQL | SQLite |
|-----------------|------------|-------|--------|
| `string` | `varchar(255)` | `varchar(255)` | `text()` |
| `number` | `doublePrecision()` | `double()` | `real()` |
| `boolean` | `boolean()` | `boolean()` | `integer({mode: 'boolean'})` |
| `bigint` | `bigint({mode: 'bigint'})` | `bigint({mode: 'bigint'})` | `blob({mode: 'bigint'})` |
| `Date` | `timestamp()` | `timestamp()` | `integer({mode: 'timestamp'})` |

#### Format Types (from `@mionkit/type-formats`)

| Format Type | PostgreSQL | MySQL | SQLite |
|-------------|------------|-------|--------|
| `StrUUID` / `StrUUIDv7` | `uuid()` | `varchar(36)` | `text()` |
| `StrEmail` | `varchar(254)` | `varchar(254)` | `text()` |
| `StrURL` | `varchar(2048)` | `varchar(2048)` | `text()` |
| `StrDomain` | `varchar(253)` | `varchar(253)` | `text()` |
| `StrIP` / `StrIPv4` / `StrIPv6` | `inet()` | `varchar(45)` | `text()` |
| `StrDateTime` | `timestamp()` | `datetime()` | `text()` |
| `StrDate` | `date()` | `date()` | `text()` |
| `StrTime` | `time()` | `time()` | `text()` |
| `NumInteger` / `NumInt32` / etc. | `integer()` | `int()` | `integer()` |
| `NumFloat` / `NumFloat32` / etc. | `doublePrecision()` | `double()` | `real()` |
| `BigIntFormat` | `bigint({mode: 'bigint'})` | `bigint({mode: 'bigint'})` | `blob({mode: 'bigint'})` |
| `StrFormat` (with maxLength) | `varchar(maxLength * buffer)` | `varchar(maxLength * buffer)` | `text()` |

Well-known formats like `StrEmail`, `StrURL`, and `StrDomain` use their standard maximum lengths without any buffer applied.

#### Complex Types

| TypeScript Type | PostgreSQL | MySQL | SQLite |
|-----------------|------------|-------|--------|
| `T[]` (array) | `jsonb()` | `json()` | `text({mode: 'json'})` |
| `{...}` (object) | `jsonb()` | `json()` | `text({mode: 'json'})` |
| `T?` (optional) | nullable | nullable | nullable |


## How To Use

#### Primary and Foreign Keys

Primary keys and foreign keys should always be defined in the tableConfig. This gives you full control over key constraints and relationships:

<code-import path="packages/examples/src/drizzle/drizzle-keys-example.ts" lang="ts" />

#### Table Config Match

The package validates that your tableConfig matches the TypeScript type:

```ts
interface User {
    id: string;
    name: string;
}

// âťŚ Error: Column "email" exists in tableConfig but not in type "User"
const users = mapPGTable<User>().build('users', {
    id: uuid('id').primaryKey(),
    email: text('email'), // This property doesn't exist in User!
});
```

#### Length Buffer

The `lengthBuffer` option multiplies the `maxLength` defined in custom string formats (`StrFormat`) to provide extra capacity for the database column. This is useful when you want to allow for some flexibility in the data length prevent resizing in the future, but still want to validate the correct length.

::note
The `lengthBuffer` only applies to custom `StrFormat` types with a `maxLength` constraint. Well-known formats like `StrEmail`, `StrURL`, and `StrDomain` use their standard maximum lengths without any buffer.
::

```ts
// Custom string format with maxLength
type Username = StrFormat<{maxLength: 50}>;

interface User {
    id: StrUUIDv7;
    username: Username;
}

// Default lengthBuffer is 1.5
const users = mapPGTable<User>().build('users');
// username with maxLength 50 â†’ varchar(75)

// Custom lengthBuffer of 2.0
const users = mapPGTable<User>({lengthBuffer: 2.0}).build('users');
// username with maxLength 50 â†’ varchar(100)
```

## Important Notes

#### Import Format Types Correctly

::info
When using format types from `@mionkit/type-formats`, you must use **regular imports** (not `import type`) for the runtime type metadata to be preserved.
::

```ts
// âś… Correct - regular import preserves metadata
import {StrUUIDv7, StrEmail} from '@mionkit/type-formats/FormatsString';

// âťŚ Wrong - type import strips metadata
import type {StrUUIDv7, StrEmail} from '@mionkit/type-formats';
```

#### Nested Objects vs Foreign Keys

Nested objects are stored as JSON. Use foreign key IDs for entity references:

```ts
// âś… Good: Profile is a value object - stored as JSON
interface User {
    id: string;
    profile: {bio: string; avatar: string};
}

// âťŚ Bad: Don't embed entire entities
interface Book {
    id: string;
    owner: User; // This stores the entire User as JSON!
}

// âś… Good: Use foreign key ID instead
interface Book {
    id: string;
    ownerId: string; // Reference by ID
}
```
