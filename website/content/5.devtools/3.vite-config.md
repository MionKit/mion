The `mionVitePlugin` from `@mionjs/devtools/vite-plugin` is the single Vite plugin behind mion's build-time features: ts-runtypes type compilation (validation & serialization), the `serverMapFrom` mapper transport, and an optional managed server for client dev/e2e builds.

## Plugin Options

```typescript
import {mionVitePlugin} from '@mionjs/devtools/vite-plugin';

mionVitePlugin({
    // ts-runtypes type compilation — rewrites route()/createValidate()/… call sites with the
    // precompiled validation & serialization functions at build time.
    // Required for mion's validation and serialization.
    runTypes?: {
        tsConfig?: string;                // Path to tsconfig.json (auto-discovered by default)
        genDir?: string;                  // Generated-output root (default: <srcDir>/__runtypes)
        emitMode?: 'code' | 'functions' | 'both'; // What generated fn entries ship (default: 'code')
        failOnError?: boolean;            // Halt the build on Error-severity diagnostics (default: true)
        allowUncheckedPatterns?: boolean; // Allow TypeFormat regex patterns RE2 cannot compile
    },

    // serverMapFrom transport between the client and server builds:
    //   the CLIENT build EMITs the inline mappers it harvested into a manifest,
    //   the SERVER build CONSUMEs that manifest (inlined into the bundle at build time).
    serverMappers?: {
        emit?: boolean | string;          // CLIENT build: write harvested mappers to this manifest path
        consume?: string | string[];      // SERVER build: manifest(s) baked in via virtual:mion/server-mappers
    },

    // Managed mion server for client dev / test / e2e builds: spawned via vite-node so the
    // client dev server has a live API to call. Awaited through serverReady.
    server?: {
        startScript: string;              // Absolute path to the server entry script (required)
        viteConfig?: string;              // Server's vite config for vite-node (auto-discovered)
        runMode?: 'childProcess';         // Only 'childProcess' is supported
        waitTimeout?: number;             // Max ms to wait for the server port (default: 30000)
        env?: Record<string, string>;     // Extra env vars for the server process
    },
});
```

::note
Only `server.runMode: 'childProcess'` is supported. The legacy `serverPureFunctions` and `aotCaches` options are still accepted but **ignored** since the ts-runtypes migration — build-time compilation is always on, and `serverMapFrom` mappers now travel through `serverMappers`.
::

### serverMappers (serverMapFrom transport)

`serverMapFrom` mappers are written in **client** code but must run on the **server**. The plugin moves them across the two builds through a small manifest:

- On the **client** build, set `serverMappers.emit` to a manifest path — the plugin harvests every inline `serverMapFrom` mapper it finds and writes it there.
- On the **server** build, set `serverMappers.consume` to that same path — the mappers are inlined into the server bundle at build time and registered through the side-effect module `virtual:mion/server-mappers` (import it once from the server entry). No `node:fs` ends up in the artifact, so it stays edge/lambda-safe.
