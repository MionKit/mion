mion provides a system for **pure functions** — functions that follow strict purity rules (no closures, no external variables, no side effects) so they can be serialized, moved between the client and server builds, and executed safely. There are two APIs, with different directions of flow:

| API | Defined in | Direction | Purpose |
|-----|-----------|-----------|---------|
| `registerMionPureFn()` | Server code | Server-side registry | Register a pure utility under a namespace so compiled functions and client→server mappers can reference it by name |
| `serverMapFrom()` | Client code | Client → Server | Map the output of one route to the input of another inside a `routesFlow`; the mapper is harvested from the client build and baked into the server |

Both follow the same **purity rules** and are processed by the mion Vite plugin at build time.

## registerMionPureFn (server-side pure functions)

`registerMionPureFn(namespace, factory)` (from `@mionjs/run-types`) registers a pure utility function on the server under a **namespace**. The factory returns the actual function, so any constants or dependencies it needs are created once at registration time:

<code-import path="packages/examples/src/introduction/pure-functions-examples.ts" lang="ts" commentStart="start:register-factory-basic" commentEnd="end:register-factory-basic" />

The namespace groups related functions so different libraries or modules can register their own without naming conflicts. Because these functions are pure, they can be embedded into the generated validation/serialization code and referenced by name — for example from a client build's `serverMapFrom` mappers.

## serverMapFrom (client → server data mapping)

`serverMapFrom()` (from `@mionjs/client`) is used in **client** code inside a `routesFlow` to map the output of one route into the input of another. The mapper function is written in the client but harvested at build time and executed on the **server**, enabling GraphQL-style data composition without extra round trips:

<code-import path="packages/examples/src/introduction/pure-functions-examples.ts" lang="ts" commentStart="start:map-from-basic" commentEnd="end:map-from-basic" />

`serverMapFrom(sourceRequest, mapperFn)` returns a reference whose `.asArg()` produces a typed placeholder that satisfies the TypeScript compiler, while at runtime the server runs the mapper to extract the actual value from the source result. See [Routes Flow](/client/client-flow) for the full data-composition guide and the `serverMappers` build wiring.

## Purity rules

Every function passed to `registerMionPureFn` or `serverMapFrom` must be **pure**:

::stylish-list
---
type: check
---
- [Only its parameters and locals:]{.text-highlighted} no closures over outer-scope or module variables
- [No side effects:]{.text-highlighted} no `this`, no `await`, no dynamic `import()`
- [No forbidden globals:]{.text-highlighted} no `eval`, `fetch`, `setTimeout`, `process`, `window`, etc.
- [Allowed globals only:]{.text-highlighted} `JSON`, `Math`, `RegExp`, `console`, and other pure built-ins are fine
::

```ts
// ✅ Pure — only uses the input parameter
serverMapFrom(order, (o) => o!.userId)

// ✅ Pure — inline computation with an allowed global
serverMapFrom(order, (o) => Math.round(o!.totalUSD * 100))

// ❌ NOT pure — references an external variable (closure)
const multiplier = 2;
serverMapFrom(order, (o) => o!.totalUSD * multiplier)
```

## Non-Vite clients

When the client is **not** built with Vite (e.g., Next.js with Turbopack), the plugin cannot harvest inline mappers. Instead, register the function on the server with `registerMionPureFn` and reference it by name from `serverMapFrom`. The [`@mionjs/no-vite-client`](./eslint-rules#mionjsno-vite-client) ESLint rule enforces that these name arguments are present and are string literals.

## ESLint integration

The `@mionjs/pure-functions` ESLint rule validates purity at lint time, catching violations before they reach the build step. See the [ESLint Rules](./eslint-rules) page for details.
