mion's Vite plugin provides **Ahead-of-Time (AOT) compilation** that pre-compiles critical runtime functions at build time. Instead of writing files to disk, AOT caches are generated as **Vite virtual modules** that get embedded directly into your server bundle.

- [Zero Disk Artifacts:]{.text-highlighted} Caches live as virtual modules — no generated files to commit or manage

## How It Works

mion uses Just-In-Time (JIT) compilation to generate validation and serialization functions at runtime. While this provides great flexibility, it has two main drawbacks:

1. **Startup overhead**: Functions are compiled when the application starts
2. **Security restrictions**: Some environments (like Cloudflare Workers) don't allow dynamic code generation

AOT compilation solves both issues by pre-compiling these functions during your Vite build process.

### The Build Pipeline

The mion Vite plugin generates AOT caches through this pipeline (using the `server` configuration):

1. **`buildStart`** — The plugin spawns a short-lived `vite-node` child process that runs your server start script with `MION_COMPILE=buildOnly` (platform adapters skip `server.listen()`).
2. **Router initialization** — In the child, `initMionRouter()` runs, registers every route, and compiles JIT functions.
3. **Cache capture** — The child IPCs the serialized caches back to the parent and exits.
4. **Virtual modules** — The parent stores the caches as `aotData` and emits them whenever a source file imports `virtual:mion-aot/caches`.
5. **Bundle embedding (build) / module evaluation (dev)** — Production builds inline the cached code into the output bundle; in dev (vite-node, middleware, or your framework's SSR) the same populated module is evaluated at runtime.

The pre-pass runs in `vite dev` and `vite build`, but only when:
- `serverConfig` is set on the plugin options,
- a source file under `serverConfig.startScript`'s dir or `serverPureFunctions.clientSrcPath` actually imports `virtual:mion-aot/`,
- the process isn't itself a pre-pass child, and
- you aren't running under vitest (`VITEST` / `NODE_ENV=test` opt out — tests typically mock the cache directly).

The `server.runMode` determines how the **live server** is managed (the pre-pass is always the same):
- **`buildOnly`**: No live server — just AOT generation. Best for production builds.
- **`childProcess`**: Spawns the live server in a child, kept alive after AOT generation. Useful when the framework runs separately from the API (e.g. Next.js + vite-node).
- **`middleware`**: mion attaches as a middleware to your existing Vite dev server (e.g. Nuxt, Vue) — no separate process.

See [Vite Configuration](./vite-config#server-modes) for details on each mode.

### Virtual Modules

The plugin generates these virtual modules that you can import in your application:

| Module | Description |
|--------|-------------|
| `virtual:mion-aot/jit-fns` | Pre-compiled JIT validation/serialization functions |
| `virtual:mion-aot/pure-fns` | Pre-compiled pure functions cache |
| `virtual:mion-aot/router-cache` | Router methods metadata cache |
| `virtual:mion-aot/caches` | Combined module — imports and registers all 3 caches |
| `virtual:mion-server-pure-fns` | Server pure functions extracted from client code |

For TypeScript support, add the virtual module type definitions to your `tsconfig.json`:

```json [tsconfig.json]
{
  "compilerOptions": {
    "types": ["@mionjs/devtools/virtual-modules"]
  }
}
```

### Cache

The plugin caches AOT results on disk (in Vite's `node_modules/.vite/` by default) so subsequent builds skip the expensive `vite-node` step when your server source hasn't changed. The cache is invalidated automatically when source files change, the devtools version updates, or AOT options change.

Set `MION_AOT_FORCE=true` to force regeneration regardless of the cache.

## Important: What Gets Compiled

::warning
**AOT only compiles routes and types that are initialized when running your start script.**
::

The AOT build process works by executing your start script and capturing all routes that get registered. This means:

- Routes not loaded by the start script will not be compiled
- Types not exposed in route/middleFn parameters or return types will not be compiled
- Dynamically created types at runtime will not be compiled

**Always populated:** The plugin pre-runs your server in a `buildStart` child to fill the caches before any user code imports the virtual module — in dev and in build alike. You don't need to wonder whether the import returned empty objects in dev: it always returns the real, compiled data.
::

### Strict AOT Mode

Passing `aotCaches` to `initMionRouter` automatically flips the router into **strict AOT mode**:

- **No run-types loading**: The heavy `@mionjs/run-types` package is NOT loaded at all.
- **Fail-fast validation**: If any route/middleFn is missing from the AOT cache, an `AOTCacheError` is thrown immediately.
- **Smaller bundles**: With proper tree-shaking, run-types is excluded from production bundles.

```typescript
import {aotCaches} from 'virtual:mion-aot/caches';
import {initMionRouter} from '@mionjs/router';

await initMionRouter(routes, {aotCaches});
// ↑ presence of aotCaches enables strict AOT mode
```

::warning
With `aotCaches` provided, ALL routes and middleFns MUST be present in the AOT cache. If any are missing, the router will throw an `AOTCacheError` instead of falling back to JIT compilation.
::

### Using AOT Caches from Other Packages

`initMionRouter({ aotCaches })` is the standard entry point. If you need to populate the global AOT registry from your own code (e.g. a standalone `@mionjs/run-types` setup with no router), import the caches and call `loadAOTCaches`:

```typescript
import {aotCaches} from 'virtual:mion-aot/caches';
import {loadAOTCaches} from '@mionjs/core';

// Manually populate @mionjs/core's global JIT / pure / router caches.
loadAOTCaches(aotCaches);
```

The plugin emits `virtual:mion-aot/caches` from any package in your monorepo as long as the Vite plugin is active in that build. No extra wiring needed.

## Secure Environments (Cloudflare Workers, etc.)

AOT compilation is essential for running mion in secure environments that restrict dynamic code execution.
Environments like Cloudflare Workers, Deno Deploy, and some enterprise environments prohibit:
- `eval()`
- `new Function()`
- Other forms of dynamic code generation

mion's JIT compilation normally uses `new Function()` to generate validation and serialization functions. AOT compilation pre-generates these functions as static code, eliminating the need for runtime code generation.
