Middleware functions are defined using `useFn` and can be executed before or after a route. middleware are function that can be chained to create the full [Execution Chain](/server/use-fns) of a route.

Middleware functions are useful when a route might require some extra data like authorization, filters or some extra processing like body parsing, logging, etc...

The first parameter is always the [Call Context](/server/routes), the rest of parameters are remote parameters that get deserialized and validated before the useFn gets executed.

## Raw Middleware functions

In case we need to access the raw or native underlying request and response, we must use a [`useRawFn`](#rawuseFndef).

These are functions that receive the `CallContext`, `RawRequest`, `RawResponse` and `RouterOptions`, but can't receive any remote parameters or return any data. 
`Raw Middleware functions` can only modify the CallContext and return or throw errors.

Raw Middleware functions are useful to extend the router's core functionality, i.e: The router internally uses a `Raw Middleware function` for body serialization.

## Execution Chain

The Execution Chain is a list of all the useFns and the route that are executed during a route call.
That is: all useFns before the route, the route itself, and all useFns after the route.

For every incoming request the Execution Chain is retrieved and all useFns and the route get executed in order.

### Execution Order

::note
The order in which Routes and Middleware functions are added to the router is important.
Middleware functions and Routes are executed in the same order they are defined (Top Down order).
::

### Middleware function's Scope

We can limit useFns to be executed only on a subset of routes.

The `userOnlyUseFn` from previous example will be executed **only** for the routes under `users` but not for routes under `pets`.
