## Features


::stylish-list
---
type: check
---
- [RPC-style routing:]{.text-highlighted} Direct method calls, no URL parsing or regex matching
- [Automatic validation:]{.text-highlighted} All parameters are validated before execution using TypeScript types
- [Automatic serialization:]{.text-highlighted} Complex types (Date, Map, Set, BigInt) serialized automatically
- [MiddleFns (middleware):]{.text-highlighted} Execute code before/after routes for auth, logging, etc.
- [Type-safe:]{.text-highlighted} Full TypeScript support with end-to-end type safety
- [Lightweight & fast:]{.text-highlighted} Simple in-memory map for route lookup
- [Nested routes:]{.text-highlighted} Organize routes in a hierarchical structure
::

**Why return errors instead of throwing?**
<br>
Returned errors are part of the function signature, giving the client full type information about possible error types and their data. The client can then handle each error type with autocompletion and type checking.
<br>

⚠️ Thrown errors are not strongly typed and should be treated as unexpected errors (not part of the Business logic).
::

## Jargon

::card-group{class="md:grid-cols-3"}
  :::card
  ---
  title: Routes
  icon: i-lucide-route
  to: /server/routes
  ---
  Methods that can be called remotely and have a URL assigned.
  :::

  :::card
  ---
  title: MiddleFns
  icon: i-lucide-link
  to: /server/middle-fns
  ---
  Auxiliary or [(middleware)]{.text-highlighted} functions that get executed before or after a route. MiddleFns can send/receive data but cannot be called directly.
  :::

  :::card
  ---
  title: Execution Chain
  icon: i-lucide-ellipsis
  to: /server/middle-fns
  ---
  Ordered list of all the methods to be executed when calling a route. Each Execution Chain can contain multiple MiddleFns but a single Route method.
  :::

  :::card
  ---
  title: Call Context
  icon: i-lucide-box
  to: /server/routes
  ---
  The context passed to every route or middleFn method, contains data like request, response, and context data. It is always the first parameter of any route or middleFn handler.
  :::

  :::card
  ---
  title: RunTypes
  icon: i-lucide-scan-text
  to: /run-types/overview
  ---
  mion uses @mionjs/run-types for automatic validation and serialization. When TypeScript gets compiled, extra bytecode is generated containing type metadata that can later be accessed at runtime and used for validation, serialization, and many other extra functionality.
  :::
::

## Powered by run-types

mion uses `@mionjs/run-types` for automatic validation and serialization. When TypeScript is compiled, type metadata is extracted and used at runtime — no schemas to define or maintain.

**What happens automatically:**

::stylish-list
---
type: check
---
- [Validation:]{.text-highlighted} All route/middleFn parameters are validated before execution
- [JSON Serialization:]{.text-highlighted} Complex types (Date, Map, Set, BigInt) are serialized correctly
- [Type Errors:]{.text-highlighted} Detailed error messages when validation fails
- [Zero Configuration:]{.text-highlighted} Works out of the box with your TypeScript types
::
